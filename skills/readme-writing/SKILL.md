# README Writing

## Purpose

Make a project understandable, usable, and worth a newcomer’s time.

## Principles

- The top of the README should show the single most important thing a newcomer
  should understand: why the project matters, what was discovered or improved,
  or what concrete problem it solves.
- Show, do not tell. Prefer screenshots, plots, examples, commands, demos,
  benchmark numbers, or artifact links over generic claims.
- Put the strongest and most important material first, notable secondary
  details later, and omit details irrelevant to readers.
- Ease of use matters most: a reader should know how to install, run, inspect,
  and reproduce the project with minimal guesswork.

## Workflow

1. Identify the target reader and the one core message the README should convey.
2. Inspect the project, artifacts, plots, and evidence before rewriting claims.
3. Structure the README in a pyramidal order:
   - project name and core value or finding
   - quick start or minimal usage path
   - evidence, examples, screenshots, plots, or results
   - installation, configuration, reproduction, and limitations
   - citation, license, acknowledgements, and secondary details
4. Improve artifact visibility: link important outputs, make plots readable and
   meaningful, and include images only when they help readers assess the work.
5. For research-like projects, include a citation block near the bottom, but not
   so far down that it is effectively hidden.

## Rules

1. Do not make unsupported claims or hide limitations.
2. Do not lead with internal development history unless that history is the
   product.
3. Keep tables sparse and evidence-oriented; do not use them as a substitute
   for clear prose.
4. Prefer one working command over a long list of possible commands.
